# API Documentation & External Services

This document provides a comprehensive overview of the APIs utilized within the Voyage project, categorized by internal REST endpoints, WebSocket events, and External Third-Party APIs.

---

## 1. Internal REST API (Backend to Frontend)

All internal API routes are prefixed with `/api`. They require a valid Bearer Token (JWT from Clerk) in the `Authorization` header.

### Trip Management (`/api/trips`)
- **`GET /`**: Returns an array of trips the authenticated user belongs to.
- **`POST /`**: Creates a new trip. Payload requires `title`, `description`, `startDate`.
- **`POST /join`**: Joins an existing trip. Payload requires `joinId`.
- **`GET /:id`**: Returns complete, populated data for a single trip.
- **`POST /:id/finalize`**: Triggers the AI summary generation.
- **`POST /:id/ai-chat`**: Payload: `{ "messages": [{ "role": "user", "content": "..." }] }`. Returns a single AI response string.

### Checkpoints (`/api/trips/:tripId/checkpoints`)
- **`POST /`**: Adds a new place to visit. Payload requires `name`, `location` (lat/lng).
- **`POST /:checkpointId/visit`**: Marks the location as visited.
- **`POST /explore`**: Payload: `{ "query": "museums in paris" }`. Returns a structured JSON array of places generated by Gemini.

### Users (`/api/users`)
- **`POST /sync`**: Syncs Clerk auth data with MongoDB. Payload: `name`, `email`, `avatar`.
- **`PUT /me/status`**: Updates the user's live status. Payload: `{ "status": "Running late" }`.

### Voice & Media
- **`GET /api/trips/:tripId/voice/token`**: Returns a signed JWT for LiveKit.
- **`POST /api/trips/:tripId/media/upload`**: Expects `multipart/form-data` with a `file` field. Returns the Cloudinary URL.

## 2. Real-time API (Socket.io Events)

The real-time communication happens via Socket.io. The server is configured in `server/socket.js`, and the client listens via `client/src/pages/TripDashboard/useTripSocket.js`.

### Client Emits (Sending Data)
- **`join_user`**: Payload: `clerkId`. Joins a personal room for global notifications.
- **`join_trip`**: Payload: `tripId`. Joins the specific trip workspace room.
- **`send_message`**: Payload: `{ tripId, senderId, content, type }`. Sent when a user chats.
- **`update_location`**: Payload: `{ tripId, userId, lat, lng }`. Emitted frequently to track live movement.
- **`update_status`**: Payload: `{ tripId, userId, status }`. Broadcasts user state changes.

### Server Emits (Receiving Data)
- **`receive_message`**: Server broadcasts the chat message (and instantly caches it in Redis) to all users in the trip room.
- **`member_location_update`**: Received by clients to update map marker positions.
- **`checkpoint_added` / `checkpoint_updated` / `checkpoint_removed`**: Received by clients to update the itinerary UI in real-time without refreshing.
- **`receive_notification`**: Server sends system-wide toasts (e.g., "User needs help!").

## 3. External APIs & Services

The application integrates heavily with external APIs to provide robust features without reinventing the wheel.

### Authentication: **Clerk**
- **Frontend Usage:** Clerk React components (`<SignIn>`, `<SignUp>`) manage the UI flow. The `useAuth()` hook provides the session token.
- **Backend Usage:** The Node SDK validates the incoming JWT on every protected route. If valid, it extracts the `userId` (`clerkId`) and attaches the corresponding MongoDB User document to the request (`req.dbUser`).

### Real-time Voice: **LiveKit**
- **Flow:** The client cannot connect directly to LiveKit without permission. The frontend calls the backend `GET /token` route. The backend uses the `livekit-server-sdk` and the hidden `LIVEKIT_API_SECRET` to sign a temporary access token. The frontend then passes this token to the `<LiveKitRoom>` component, which establishes the WebRTC connection directly to the LiveKit cloud.

### Artificial Intelligence: **Google Gemini & Groq**
- **Flow:** Both services are used server-side only to protect API keys.
- **Gemini (`@google/genai`):** Used primarily for the 'Finalize Trip' feature because of its advanced capabilities (Gemini 2.5 Flash) and ability to ingest Images (Vision).
- **Groq (`groq-sdk`):** Used as the primary engine for the AI Chatbox and as a fallback. Groq runs Llama 3.1 models at incredibly high speeds, making it perfect for real-time chat interactions.

### Media Storage: **Cloudinary**
- **Flow:** When a user uploads an image, the frontend sends it to the backend. The backend uses `multer-storage-cloudinary` to intercept the file buffer and stream it directly to Cloudinary's servers. Cloudinary returns a secure URL, which the backend saves to MongoDB. This prevents the Node server from having to store images locally.
